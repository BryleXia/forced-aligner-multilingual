  ---
  核心发现：9号有严重的"分块跳跃"问题

  9号遇到长音频时会把文本和时间各切成几块对齐，但分块逻辑有根本缺陷——它假设每段
  时间里台词数量相同，但实际上前几分钟台词密集、后面稀疏，导致每个分块边界处时间
  轴突然跳跃：

  ┌───────────────┬───────────┬───────────┬─────────────┐
  │     位置      │ 8号时间戳 │ 9号时间戳 │    问题     │
  ├───────────────┼───────────┼───────────┼─────────────┤
  │ seg001 第18行 │ 03:37     │ 05:00     │ 跳了1分23秒 │
  ├───────────────┼───────────┼───────────┼─────────────┤
  │ seg001 第35行 │ 07:43     │ 10:00     │ 跳了2分17秒 │
  ├───────────────┼───────────┼───────────┼─────────────┤
  │ seg002 第20行 │ 04:31     │ 05:00     │ 跳了29秒    │
  ├───────────────┼───────────┼───────────┼─────────────┤
  │ seg002 第39行 │ 08:11     │ 10:00     │ 跳了1分49秒 │
  ├───────────────┼───────────┼───────────┼─────────────┤
  │ seg002 第58行 │ 11:52     │ 15:00     │ 跳了3分8秒  │
  └───────────────┴───────────┴───────────┴─────────────┘

  但在跳跃之前的行，9号与8号几乎完全相同（差距在毫秒级，不影响体验）。

  所以结论：
  - 9号在分块边界前：≈ 8号质量
  - 9号在分块边界处及之后：完全失败，比8号差得多

  ---
  原因与修复方向

  问题出在我写的 MAX_CHUNK_DURATION = 300
  这个参数上。你的5段音频最长只有约16分钟（955秒），完全不需要分块。只要把这个阈
  值调大，让所有音频都作为整体处理，跳跃问题就消失了。

  需要把脚本里这一行改掉：

  MAX_CHUNK_DURATION = 300.0   ← 当前值，导致分块
  MAX_CHUNK_DURATION = 1800.0  ← 改成这个，30分钟，5段音频全都不会触发分块

❯ 我想知道，为什么九号修复版会跑这么久？我用的可是RTX PRO 6000啊！              

● 很好的问题。原因在于强制对齐（forced alignment）的计算特性，和 GPU
  算力关系不大。

  ---
  为什么慢？

  8号的流程：
  Whisper 把16分钟音频切成几十个小段（每段几秒），每次 whisperx.align()
  只处理几秒的短片段 → 快

  9号修复版的流程：
  整段16分钟音频作为一个片段喂给 whisperx.align() → 慢

  ---
  CTC 强制对齐的计算性质

  wav2vec2 做强制对齐的核心算法是 CTC Viterbi 动态规划，它本质上是：

  ▎ 对每一帧音频 × 每一个文本字符，填一张巨大的二维表格

  这张表的大小 = 音频帧数 × 文本字符数

  - 16分钟音频 ≈ 47,000帧
  - 1400个词 × 平均6个字母 ≈ 8,400个字符
  - 表格大小 ≈ 4亿个格子，而且必须按顺序填，无法并行

  RTX PRO 6000 的强项是大规模并行矩阵运算（比如 Transformer
  的注意力机制），但这里的动态规划是一步步递推的，GPU
  并行优势发挥不出来，大部分时间消耗在 CPU 上。

  ---
  对比

  ┌────────────────────┬───────────────────────┬─────────────────────────┐
  │                    │          8号          │        9号修复版        │
  ├────────────────────┼───────────────────────┼─────────────────────────┤
  │ 每次对齐的音频长度 │ 几秒                  │ 16分钟                  │
  ├────────────────────┼───────────────────────┼─────────────────────────┤
  │ GPU 利用率         │ wav2vec2 编码器能用上 │ 编码器快，但 CTC 解码慢 │
  ├────────────────────┼───────────────────────┼─────────────────────────┤
  │ 时间               │ 快                    │ 慢很多                  │
  └────────────────────┴───────────────────────┴─────────────────────────┘

  总结：不是显卡不够强，是这个算法天生不适合处理超长单段。如果速度是问题，可以考
  虑换回分块策略，但用更聪明的分块方式（按静音点切，而不是按等时间切）。
